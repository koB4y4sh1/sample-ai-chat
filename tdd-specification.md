# Zenith AI Chat TDD 仕様

## 目的

この文書は、Zenith AI Chat の日常開発をテスト駆動で進めるための共通手順を定義する。
対象は `web`、`agent`、`mcp` の各変更であり、実装前に期待挙動を固定し、変更の完了条件を明確にすることを目的とする。

## 適用範囲

- 新規機能追加
- 既存挙動の修正
- API 契約の変更
- 回帰バグ修正

単なる文言修正や import smoke test のみで十分な scaffold 調整は、必ずしも完全な TDD サイクルを要求しない。

## 基本原則

- 実装前に期待挙動を文章とテストで固定する。
- 1 回のサイクルでは 1 つの失敗理由だけを作る。
- テストは実装詳細ではなく契約と観測可能な挙動を表現する。
- Green になった直後に不要な重複と命名の不整合を解消する。
- リファクタリングは Green を維持したまま行う。
- 仕様変更はテスト、実装、関連ドキュメントを同時に更新する。

## 開発サイクル

標準サイクルは以下とする。

1. 変更対象の要求を 1 文で書く。
2. 受け入れ条件を 1 つ以上に分解する。
3. 最小の失敗テストを書く。
4. 失敗を確認する。
5. テストを通す最小実装を加える。
6. 同じテストを再実行して Green を確認する。
7. 命名、重複、責務分離だけを対象にリファクタリングする。
8. 関連する狭い範囲のテストと品質ゲートを回す。

## 要求の書き方

実装に入る前に、変更要求を次の形式で残す。

```text
変更要求:
- ユーザーとして、送信失敗時に再試行できること。

受け入れ条件:
- 送信 API が失敗したときエラーメッセージが表示される。
- 再試行操作で同じ入力が再送される。
- 成功時はエラー表示が消える。
```

Issue、PR 説明、作業メモ、または設計メモのいずれかに残せばよいが、テスト名から読み取れる粒度まで分解すること。

## 層ごとのテスト方針

### web

- 画面の重要挙動は UI テストで表現する。
- 単なるスタイル差分ではなく、ユーザー操作と画面状態の変化を確認する。
- BFF 通信はモックまたは契約に沿ったスタブで置き換える。
- コンポーネント内部の state 実装には依存しない。

### CopilotKit

- CopilotKit 連携を変更する場合は、runtime 内部ではなく外側から観測できる契約を Red で固定する。
- 対象契約は、`/api/copilotkit` の request/response、agent context、provider/model による endpoint 選択、frontend tool の登録、tool render に渡る入力、チャットUIの表示変化とする。
- CopilotKit Runtime は境界として扱い、テストでは mock または契約に沿った stub で置き換えてよい。
- Generative UI の追加や変更では、tool 名、入力 schema、表示結果、失敗時の扱いを受け入れ条件に含める。

### web BFF

- エンドポイント単位で request、response、エラー形式を固定する。
- 入力検証、下流呼び出し失敗、正常系を最低限含める。
- FastAPI ルートが増えたら API 契約テストを優先する。
- 下流依存はテストダブルで隔離し、BFF の責務に集中する。

### agent

- エージェントの意思決定そのものではなく、入出力契約、状態遷移、ツール選択条件を検証する。
- 外部依存はスタブ化し、決定不能な runtime 実装に引きずられないようにする。
- 会話状態の更新、失敗時の扱い、タイムアウトを優先的にテストする。

### mcp

- tool の入力スキーマ、出力スキーマ、エラー分類を契約として固定する。
- 副作用がある tool は入力検証、タイムアウト、失敗時の戻り値を明示的にテストする。
- agent 側の都合ではなく、tool 単体の契約を守る。

## テストの粒度

- 最初の 1 本はもっとも外側の観測可能な振る舞いから書く。
- 失敗原因の切り分けが難しい場合だけ 1 段内側のテストを追加する。
- private 関数のためだけのテストは作らない。
- import smoke test は scaffold 段階の package に限定する。

## Red の定義

Red は次を満たす状態とする。

- 追加したテストが現在の実装で失敗する。
- 失敗理由が今回の要求に対応している。
- タイポや環境不備ではなく、期待挙動との不一致で落ちている。

実装前に失敗確認を省略してはならない。ただし既存の failing test がある場合は、対象範囲を絞って今回追加したテストの失敗を確認する。

## Green の定義

Green は次を満たす状態とする。

- 今回追加または更新した対象テストが通る。
- 変更範囲に関連する既存テストが通る。
- `pnpm run check` に含まれる品質ゲートに矛盾しない。

Green の時点では、実装が最終形である必要はない。まず挙動を固定し、その後に整理する。

## Refactor の定義

Refactor では次だけを行う。

- 重複の削減
- 命名の改善
- 責務分離の改善
- テストデータや補助関数の整理

Refactor では仕様を増やさない。仕様を増やす場合は新しい Red からやり直す。

## 完了条件

1 つの変更が完了したと見なす条件は以下。

- 受け入れ条件に対応するテストが存在する。
- 追加したテストが Green である。
- 関連ドキュメントが必要に応じて更新されている。
- フォーマット、lint、型検査に通る。
- PR やコミット説明から、何を保証した変更か読み取れる。

## 品質ゲートエラーの追従

- Ruff、Biome、`tsc`、mypy の診断は、TDD サイクルの Green 条件を阻害する作業項目として扱う。
- 今回変更で発生または露出した診断は、対象テストが Green になった後に解消し、同じコマンドを再実行して確認する。
- 自動修正後も残る診断は、テスト失敗と同じく原因を 1 件ずつ切り分ける。
- unrelated な既存診断が残る場合は、作業範囲外として記録し、全体品質ゲートが通ったとは扱わない。
- rule の ignore 追加や診断抑制は、対象範囲と理由を明示できる場合だけ許可する。

## 推奨コマンド

作業中は対象範囲を狭く、区切りで全体チェックを実行する。

```bash
pnpm run test
pnpm run check
uv run pre-commit run --all-files
```

フロント開発中は別ターミナルで以下を維持してよい。

```bash
pnpm run dev:web
```

## 変更タイプ別の開始点

### バグ修正

1. 再現テストを書く。
2. 失敗を確認する。
3. 最小修正を入れる。
4. 同種の回帰がないか近傍ケースを追加する。

### 新機能

1. 受け入れ条件を 2 から 4 個に分ける。
2. 1 条件ずつ Red から進める。
3. 条件をまとめて一気に実装しない。

### 契約変更

1. 契約テストを先に更新する。
2. 利用側のテストを更新する。
3. 実装を追従させる。
4. 仕様書を更新する。

## やってはいけないこと

- 実装を先に書いてから後付けでテスト名だけ整えること
- 複数要求を 1 本の曖昧なテストにまとめること
- 内部実装を固定しすぎてリファクタリング不能にすること
- 失敗確認をせずにテストを書いたことにすること
- 既存 failing test を無視して全体 Green と誤認すること

## このリポジトリでの最低運用

現時点では `agent`、`mcp` は scaffold 中であるため、初期フェーズでは import smoke test だけの箇所が存在してよい。
ただし API ルート、agent 状態遷移、MCP tool 実装を追加した時点で、その変更は smoke test のみでは完了扱いにしない。

## 仕様更新ルール

以下を変更した場合は、この文書または関連仕様を更新する。

- テスト戦略
- 完了条件
- 標準コマンド
- 契約変更時の手順
- 層ごとのテスト責務
