MCP Theme Decision Log

MCP テーマ選定メモ

要点: MCP に載せるかは「再利用できる tool 境界か」「ユーザー入力だけで価値が出るか」「Generative UI と見た目が壊れないか」で切る。業務テーマは監査・個人情報コストが重くなりやすい。

本文はテーマごとの判断ログと運用上の注意が続く。

MCP のテーマを決めるときに議論した内容を、後から設計判断を読み返せるように残す。コントラストや配色だけでなく、MCP Apps への移植しやすさやデータ利用のリスクまで含めて整理する。

前提となる前提

評価軸

テーマ選定では次の 4 軸を並べる。

  • 業務上「説明しやすい」こと
  • 長く触っても疲れないこと
  • MCP App と Generative UI で見た目が破綻しないこと
  • 将来の拡張で無理なくモックや別ブランドに差し替えられること

設計の大原則

外部システムと結びつく処理は安易に MCP に寄せない。再利用可能な tool 境界として切り出せるかで判断する。アプリ内の state を細かい utility で埋める実装は、原則 MCP にしない。

MCP に載せるテーマの考え方

向いている

  • 比較・レビュー・承認・進捗のように状態と次アクションがある UI
  • ユーザー入力データだけで最初の結果が出せる
  • カード、表、チェックリスト、タイムラインなど UI の見せ場がある
  • 原則として外部サービス連携を追加しても責務が崩れにくい

向かない

  • 細かい plan や長文メモだけで業務が完結する、tool 境界が曖昧
  • 入力を細かく刻んで検証するだけで、アプリ内関数で足りる
  • 外部データ利用許諾や確実な監査証跡が最初から重い
  • 個人情報や機微情報など、コストとガバナンスが先に来る

読みやすさと設計

テーマ 強み 負担 設計
ライト基調・低彩 UI・比較・MCP App 連携が多い スクロール長文・多用時の疲労・再利用時のコストが軽い 強調色が弱くアクティブ状態では視認が落ちる テーマとしては淡色寄り。アクティブ状態は別トークンで補う
業務オペレーション寄り 推移や履歴で文脈が見える 依存データ・個人情報・監査ログが重い 業務専用テーマを選ぶ。レビュー用途ではログ設計もセットで決める
イベント・レビュー寄り 比較 UI を組みやすくチャットから繋ぎやすい 外部データ探索と許諾確認の工程が増えやすい アイデアとして薄色寄り。長文テーマとしては判断が重い

ライト基調・レビュー寄りはアイデアとして強いが、MCP App とブランドが競合しやすい。よってスコープでは UI よりデータ探索と利用許諾のコストが勝ちやすい。

よくある落とし穴

  • 全体トークンが明暗ワンスクリーン前提だと強調が弱い
  • 条件付き・段階開示・匿名化は業務ドキュメント以外の許諾設計が先
  • 実務レビューと個人情報・監査ログが同居すると運用コストが跳ねる
  • 機密と API の粒度がサイトごとにバラバラだと現場コストが重い

業務寄りテーマの埋め込み

業務種別は固定ではないが、実務レビューに寄せると強調が大きい。設計を誤ると業務オペレーションそのものではなく「業務サマリー」「個人情報」「監査ログ」にモックが寄りやすい。

比較に強い業務が良い理由

  • 推移の長期運用
  • 仕様書のドリルダウン
  • 契約や変更履歴の抽出
  • 複数拠点業務の横断整理

選ぶべきでない領域

  • 個人ごとの恣意的な業務要件
  • 調達や調査目的の機微データ
  • そのまま業務 UI に載せる監査ログの丸写し

運用で選んだ active テーマ

document_review

既存資料を踏まえた文章校正・体裁調整・ファイル入力や業務レビュー連携を想定する。

listing_assist

検索ではなく棚卸し業務に寄せることで、一覧・見積・チェックリストなど MCP App の見せ場が立ちやすい。

quote_compare

複数見積の比較や業務判断が多く、表・リスク・ライクライン・運用理由 UI が組みやすい。外部データ依存も検討しやすい。

submission_pack

提出・送付はチェックリストと状態更新が主で、MCP に載せる値と MCP App の境界が説明しやすい。

あとから設計に使うルール

  1. レビュー寄りテーマを選ぶときは、最初にデータ探索と規約遵守のコストを確認する。
  2. 外部システムと結ぶなら MCP にしない前に tool 境界として説明できるかを見る。
  3. ユーザー入力データだけで成果が出るテーマを優先する。
  4. Generative UI の見せ場が表・タイムライン・チェックリスト・カードに落ちるか確認する。
  5. 業務・監査・個人情報は「許諾」「処理」「監査」に分けて語る。

運用でテーマを閉じる条件

  • ライトとイベント寄りは機密やデータ・API の粒度が荒いとテーマ維持が難しい。
  • 業務寄りは業務オペレーションではなく資料レビュー用途で設計が固まったら閉じやすい。
  • レビュー寄りテーマを閉じるときも、まずモックで UI と tool surface を揃えてから本番データ連携を語る。
